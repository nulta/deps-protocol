# 홈서버 개발자들을 위한 안내서

## 홈서버의 역할

[DEPS 소개](../README.md)에서는 홈서버의 역할을 아래와 같이 정의하고 있습니다.

- 문제증표를 저장하고 관리한다.
- 핸들네임을 부여할 수 있다.
- 이사 기능을 제공한다.

다만, 사용성을 생각해보면 실제로는 홈서버가 더 다양한 기능을 제공하게 됩니다. 이를테면...

- 제출한 소스 코드 저장
- 문제서버와 상호작용하는 UI 제공
    - "문제 목록" 페이지
    - "문제 제출" 페이지
    - "제출 결과 확인" 페이지 등
- 비밀키 보관
    - "회원가입" 기능 (즉, 아이덴티티 생성 기능)
    - "로그인" 기능 (저장하던 비밀키를 불러오는 기능)

즉, 표준에 나와 있는 *문제서버 API*들은 사용자가 직접 손댈 게 아니고 홈서버(의 프론트엔드)가 대신 처리해줘야 하는 겁니다. 이상적인 흐름이라면, 사용자는 그냥 자기 홈서버에 로그인해서 문제만 풀면 되는 것입니다.

## 비밀키 보관

그런데 표준 문서를 읽어보셨다면 **비밀키 보관** 기능에서 의아함을 가지게 될 겁니다. 표준 API는 사용자가 비밀키를 직접 관리하는 것을 전제로 설계되어 있었으니까요. 하지만 실제 구현에서는 다르게 접근할 수 있습니다.

이 부분은 홈서버 구현체의 설계 철학에 따라 아래 셋 중 하나로 접근하게 됩니다.

### 1) 비밀키를 사용자가 알아서 보관하게 하기

가장 기본적인 구조입니다. 홈서버 API는 모두 이를 전제로 설계되어 있습니다. 사실 다른 방식을 채택하더라도, API 정의상 이 구조는 여전히 지원해야 합니다.

사용자를 전적으로 믿는 구조입니다. 전적으로요.

- 장점:
    - **리스크가 적습니다.** 홈서버 DB가 통으로 털려도 애초에 별 민감한 정보가 없습니다.
    - 구현이 간단합니다. 로그인 API도 필요없습니다.
- 단점:
    - **완~전 번거롭습니다.** 애초에 비밀키 직접 관리하는 거 엄청 거추장스럽고 손도 많이 가는 일입니다. 동기화는 어쩌죠? 잃어버리기라도 하면 어째요?
    - 그리고 사용자는 생각보다 정보를 잘 잃어버립니다. 사용자가 자기 비밀키를 실수로 직접 어디다가 올려버린다면 수습할 방법이 없습니다.

여러분의 타깃 유저층이 이런 번거로움을 감수할 만큼 기술적으로 능숙하며 정보보안 지식이 많은 사람들이라면 이것만 구현하셔도 좋습니다. 하지만 일반 사용자들을 타깃으로 하는 서비스라면, 음, 귀찮아서 아무도 안 쓰지 않을까요?

### 2) 홈서버가 비밀키를 그대로 관리해주기

여기서부터 우리가 아는 전통적인 서비스들과 동일한 경험을 제공할 수 있습니다.

- 장점:
    - **편리하고 익숙합니다.** 그냥 로그인/회원가입해서 문제 풀면 됩니다. 참 쉽죠?
    - 비밀번호를 잊어버려도 <비밀번호 재설정> 같은 거 만들면 됩니다.
- 단점:
    - **리스크가 큽니다.** 만에 하나 홈서버 DB가 털린다면 즉각 초대형 사고가 됩니다!! 비밀번호와 달리 비밀키는 바꿀 방법이 없습니다!!
    - 홈서버 운영자의 윤리관을 전적으로 신뢰해야만 합니다. 물론 대부분의 운영자는 선한 사람들이겠지만 사람 일은 모르는 거니까요.

**털리지만 않으면** 가장 쉽고 간단하게 쓸 수 있습니다. **털리지만 않는다면요**.

아, 이사 기능 구현할 때 비밀키 내보내는 기능도 따로 만들어주셔야 합니다. 잊지 마세요.

### 3) 홈서버가 비밀키를 E2EE 암호화해서 관리하기

1번과 2번의 절충안입니다. **암복호화 과정은 전부 클라이언트에서 처리하셔야 합니다.** 즉, 서버는 평문 비밀키를 볼 수 없어야 합니다. 이런 방식을 E2EE(End-to-End Encryption)라고 부릅니다.

이 때 "비밀키를 다시 암호화하는 키"는 주로 사용자의 로그인 비밀번호로부터 PBKDF 같은 걸로 도출해서 쓰게 됩니다. 그게 간단하니까요. 아래의 장단점도 그걸 기준으로 합니다.

- 장점:
    - **충분히 편리합니다.** 그냥 로그인해서 문제 풀면 됩니다.
    - **충분히 안전합니다.** 애초에 서버가 사용자들의 비밀키를 볼 수 없기 때문에 DB가 털려도 비밀키는 지킬 수 있습니다. (로그인 비밀번호가 안전하다면요)
- 단점:
    - **암호학의 세계에 오신 것을 환영합니다.** 안전한 E2EE 구현은 생각보다 까다롭습니다. 지식이 함께하길 바랍니다.
    - 프론트엔드 구현 난이도가 약간 상승합니다. 대신 백엔드는 암호화된 키 보관 정도만 하면 됩니다.
    - 비밀번호 기반 로그인만 사용할 수 있습니다. 소셜 로그인이나 이메일 로그인은 구조적으로 지원하기 어렵습니다.
    - 사용자가 비밀번호를 잊어버리면 비밀키를 되찾을 방법이 없습니다. 후술할 키 백업 흐름을 만드는 것을 추천합니다.

Keybase, Bitwarden 같은 서비스들이 이런 방식을 채택하고 있습니다. 참고하세요.

### 니모닉을 이용한 비밀키 백업

DEPS 생태계에 접근하려면 아이덴티티 공개키/비밀키 쌍을 어디선가 만들긴 해야 합니다. 보통 이 역할을 홈서버가 맡게 될 것입니다.

이 때, **니모닉 코드**를 활용해서 비밀키를 백업해둘 수 있다면 정말 좋을 겁니다. 대강 아래와 유사한 흐름이 될 것입니다.

1. 키 생성용 랜덤 엔트로피를 만든다.
2. BIP-39에서 정의한 니모닉 코드로 엔트로피를 인코딩한다.
    - 대강 `paper equal hole poverty next retreat inform repair fine clip slogan extend` 같이 사람이 읽기 쉬운 형태가 됩니다.
3. 니모닉 코드를 사용자에게 보여주면서, 백업하라고 안내한다.
    - 이제부터 이 코드는 비밀키와 동등한 정보입니다.
4. 엔트로피를 이용해 Ed25519 키 쌍을 생성한다.

이렇게 키를 백업해두면 서버가 갑자기 예고 없이 불타버려도 사용자는 아이덴티티를 복구할 수 있습니다.

물론 니모닉이 유출되면 비밀키도 같이 유출되니, 안전하게 보관하라고 당부해주세요.
